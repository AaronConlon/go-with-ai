# Go With AI

这是一个用于学习 Go 开发、沉淀工程知识和推进项目实践的 Markdown 文档仓库。

当前结构分为两部分：

- `background/`：学习背景、研究结论、路线设计和项目方向沉淀。
- `practice/`：从零开始推进 HN 摘要推送 Agent 的实战目录。
- `website/`：基于 VitePress 的静态文档站点，用于持续记录 Go 学习笔记、知识卡片和项目实践。
- `hn-agent/`：Hacker News 摘要推送 Agent 的 Go 实战项目目录。这个目录由学习者在阶段一创建。

## 快速开始

进入静态站点目录：

```bash
cd website
npm install
npm run dev
```

构建静态站点：

```bash
cd website
npm run build
```

预览构建结果：

```bash
cd website
npm run preview
```

## 推荐实现代码块约定

文档中如果给出可以直接落到某个文件里的“推荐实现”，代码块第一行必须用对应语言的注释写明完整的仓库相对路径和文件名，例如：

```go
// hn-agent/internal/hn/fetch_batch.go
package hn
```

这样读者先知道这段代码应该放在哪里，再开始读具体实现。普通语法片段、命令片段和概念演示不需要强行写路径。

## 完整代码栏目约定

文档站有独立的完整代码栏目：

```text
website/code/
website/en/code/
```

每个阶段一页，保存该阶段新增或重点修改的完整文件，并包含对应测试文件。阶段尚未实现时保留占位页，后续完成实现后再补全代码。

完整代码栏目里的代码是学习版代码，不追求生产代码的最少注释。前三阶段尤其要在代码内部保留完整备注，解释关键语法、标准库、测试意图、错误处理和并发机制。

第一次出现的复杂知识点要在出现处解释，尤其是新标准库函数、新语法、新测试写法、新并发原语和不容易从名字看懂的工具函数。后续重复出现时可以简化，但第一次不能跳过。

如果示例里使用 `_` 忽略返回值，例如 `_, _ = fmt.Fprint(...)`，也要解释为什么忽略。Go 允许直接调用函数并丢弃返回值，但 `_, _ =` 更明确地表示“我知道这里有返回值，并且有意忽略”。

遇到 Go 字面量语法错误时，要解释语言规则本身。比如字符串使用双引号或反引号，单引号表示 `rune`；`gofmt` 只格式化已经能解析的 Go 代码，不负责把语法错误自动改成另一种可能改变语义的写法。

测试输出里第一次出现格式化动词时也要解释，例如 `%d` 输出十进制整数，`%v` 输出默认格式，`%#v` 输出更接近 Go 语法的详细格式，适合调试复合值。

## Go 版本约定

当前学习和文档说明以 Go 1.27 为基准。

Go 版本会影响部分语言语义、工具链行为和文档判断，例如 `for range` 循环变量捕获、`go.mod` 中的 `go` directive、依赖整理和 linter 提示。后续更新学习记录或解释代码时，需要明确区分“当前 Go 1.27 行为”和“旧版本教程里的历史写法”。

因此 Go 相关命令默认在 `hn-agent/` 内执行，例如：

```bash
cd hn-agent
go test ./...
go run ./cmd/hnctl
```
